# Индивидуальный отчёт по практике

**Тема проекта:** веб‑приложение для учёта личных финансов («Бурмалда Finance»)
**Период практики:** 4 недели
**Роль:** разработчик (полный цикл: анализ → проектирование → реализация →
тестирование → развёртывание)

---

## 1. Описание проекта и личного вклада

В рамках практики самостоятельно разработан полнофункциональный сервис учёта
личных финансов. Приложение позволяет вести доходы и расходы по категориям,
задавать месячные бюджеты с контролем лимитов и получать аналитику (сводки,
разбивку по категориям, тренд по месяцам).

Личный вклад охватывает все части проекта:

**Анализ и проектирование (1‑я неделя).**
- Сформулированы функциональные (FR‑1…FR‑14) и нефункциональные (NFR‑1…NFR‑8)
  требования, написаны пользовательские истории с критериями приёмки.
- Выбран и обоснован технологический стек (FastAPI, SQLAlchemy, SQLite,
  Pydantic, pytest).
- Спроектирована слоистая архитектура и подготовлены UML‑диаграммы (use case,
  классов, две sequence‑диаграммы) и ER‑диаграмма базы данных.

**Разработка (2‑3‑я недели).**
- Реализован бэкенд по слоям `routers → services → repositories → models` с
  внедрением зависимостей и единой обработкой ошибок.
- Описаны Pydantic‑схемы с валидацией входных данных.
- Спроектировано и задокументировано REST API (Swagger/OpenAPI «из коробки»).
- Разработан адаптивный веб‑интерфейс на чистом HTML/CSS/JS.
- Написан скрипт наполнения БД тестовыми данными.

**Тестирование и развёртывание (4‑я неделя).**
- Написаны 44 теста (модульные + интеграционные), все проходят.
- Проведено ручное тестирование сценариев, найденные дефекты исправлены.
- Подготовлены `Dockerfile`, CI‑пайплайн (GitHub Actions) и инструкция по
  развёртыванию.
- Оформлена документация: ТЗ, диаграммы, справочник API, отчёт о тестировании.

## 2. Примечание об организации работы

Задание предполагает в том числе командную работу (группы по 3–4 человека).
В данном случае проект выполнен индивидуально, поэтому соответствующие
командные артефакты (распределение ролей, протоколы встреч) заменены
самостоятельным прохождением всех ролей. Применены практики, имитирующие
командный процесс: ведение списка задач по этапам, осмысленные коммиты,
разделение кода на независимые слои (что упростило бы параллельную работу
в реальной команде).

## 3. Применённые инженерные практики

- **SOLID.** SRP — разделение на слои; DIP — сервисы зависят от репозиториев
  через конструктор, а не от конкретной СУБД; репозитории работают каждый со
  своей сущностью.
- **DRY.** Общая логика (границы месяца, агрегаты, преобразование ошибок)
  вынесена в переиспользуемые функции/обработчики.
- **KISS.** Минимум зависимостей, простая конфигурация через переменные
  окружения, SQLite вместо тяжёлого сервера БД.
- **Валидация и обработка ошибок.** Все входные данные проверяются Pydantic;
  доменные исключения транслируются в корректные HTTP‑коды и единый формат.
- **Тестируемость.** Изолированные фикстуры БД, подмена зависимостей FastAPI.

## 4. Возникшие сложности и их решение

| Сложность | Решение |
|-----------|---------|
| Несоответствие типа операции и типа категории могло привести к «грязным» данным | Введено бизнес‑правило в сервисе с проверкой и понятной ошибкой `422`; покрыто тестами |
| Расчёт исполнения бюджета требует агрегировать траты по категории за месяц | Реализованы агрегаты на стороне БД (`SUM ... GROUP BY`), вынесены в репозиторий |
| Устаревшие API фреймворка (событие `startup`, константа статуса 422) выдавали предупреждения | Переход на `lifespan`, использование числового кода `422`; зафиксировано в отчёте о тестировании |
| Деньги как `float` дают погрешность | Округление до 2 знаков на входе; ограничение задокументировано как осознанный компромисс с рекомендацией `Decimal` для продакшена |
| Извлечение месяца для тренда зависит от СУБД | Использован `strftime` (SQLite); особенность отмечена в документации |

## 5. Выводы и предложения по улучшению

**Выводы.** Пройден полный цикл разработки ПО. Слоистая архитектура и принципы
SOLID/DRY/KISS заметно упростили внесение изменений и написание тестов:
бизнес‑логика проверяется без HTTP, а API — через единый клиент. Автогенерация
Swagger и валидация Pydantic сократили объём ручного кода и количество ошибок.

**Что можно улучшить в дальнейшем:**
1. Добавить аутентификацию и многопользовательский режим (сущность `User`).
2. Перейти на PostgreSQL и миграции (Alembic) для продакшена.
3. Перейти на тип `Decimal`/целые копейки для финансовой точности.
4. Добавить графики динамики на фронтенде и экспорт данных в CSV.
5. Расширить CI до CD (автоматическая публикация Docker‑образа и деплой).

**Итог.** Поставленные цели практики достигнуты: разработан, протестирован и
подготовлен к развёртыванию работающий сервис с полным комплектом документации.

# Техническое задание (ТЗ)

**Проект:** «Бурмалда Finance» — веб‑приложение для учёта личных финансов
**Версия:** 1.0
**Дата:** 2026‑06‑13

---

## 1. Назначение и цели

Сервис предназначен для ведения учёта личных доходов и расходов, контроля
бюджета и анализа структуры трат. Пользователь может:

- фиксировать денежные операции (доходы и расходы) по категориям;
- задавать месячные бюджеты (лимиты расходов) и отслеживать их исполнение;
- получать аналитику: сводку за период, разбивку по категориям, тренд по месяцам.

**Цель** — дать наглядный инструмент для контроля личного бюджета и
продемонстрировать полный цикл разработки ПО (анализ → проектирование →
реализация → тестирование → развёртывание).

## 2. Глоссарий

| Термин | Определение |
|--------|-------------|
| Операция (Transaction) | Запись о движении денег: сумма, тип, категория, дата |
| Категория (Category) | Группа операций одного типа (например, «Продукты») |
| Тип | `income` (доход) или `expense` (расход) |
| Бюджет (Budget) | Лимит расходов по категории на конкретный месяц |
| Баланс | Разница «доходы − расходы» за период |

## 3. Функциональные требования

| ID | Требование |
|----|-----------|
| FR‑1 | Создание, просмотр, изменение и удаление **категорий**. |
| FR‑2 | Имя категории уникально в пределах своего типа. |
| FR‑3 | Создание, просмотр, изменение и удаление **операций**. |
| FR‑4 | Тип операции должен совпадать с типом её категории. |
| FR‑5 | Сумма операции — положительное число с точностью до 2 знаков. |
| FR‑6 | Фильтрация операций по типу, категории и периоду (год/месяц). |
| FR‑7 | Создание и изменение **бюджета** (лимита) по категории расходов на месяц. |
| FR‑8 | Бюджет допустим только для категорий расходов; на один месяц — один бюджет на категорию. |
| FR‑9 | Расчёт исполнения бюджета: потрачено, остаток, процент, признак перерасхода. |
| FR‑10 | Аналитика: сводка (доходы/расходы/баланс) за месяц или год. |
| FR‑11 | Аналитика: разбивка суммы по категориям за период. |
| FR‑12 | Аналитика: помесячный тренд доходов и расходов за год. |
| FR‑13 | Веб‑интерфейс для всех основных операций. |
| FR‑14 | Автоматическая документация API (Swagger UI / ReDoc). |

## 5. Пользовательские сценарии (User Stories)

- **US‑1.** *Как пользователь, я хочу добавить расход с суммой, категорией и
  датой, чтобы он учитывался в балансе.*
  Критерий приёмки: после добавления операция видна в списке, сводка за период
  пересчитана.

- **US‑2.** *Как пользователь, я хочу создавать свои категории доходов и
  расходов, чтобы группировать операции по смыслу.*
  Критерий приёмки: новая категория появляется в выпадающем списке формы.

- **US‑3.** *Как пользователь, я хочу видеть доходы, расходы и баланс за месяц,
  чтобы понимать своё финансовое состояние.*
  Критерий приёмки: карточки сводки показывают корректные суммы за выбранный период.

- **US‑4.** *Как пользователь, я хочу задать лимит расходов по категории на
  месяц и видеть, сколько уже потрачено, чтобы не выходить за бюджет.*
  Критерий приёмки: индикатор показывает процент использования и подсвечивает перерасход.

- **US‑5.** *Как пользователь, я хочу видеть, на какие категории уходит больше
  всего денег, чтобы оптимизировать траты.*
  Критерий приёмки: разбивка по категориям отсортирована по убыванию суммы.

- **US‑6.** *Как пользователь, я хочу фильтровать операции по типу и периоду,
  чтобы быстро находить нужные записи.*
  Критерий приёмки: список обновляется согласно выбранным фильтрам.

- **US‑7.** *Как разработчик‑интегратор, я хочу иметь документацию API, чтобы
  обращаться к сервису программно.*
  Критерий приёмки: доступны Swagger UI (`/docs`) и схема OpenAPI (`/openapi.json`).

## 6. Технологический стек и обоснование

| Технология | Обоснование выбора |
|------------|--------------------|
| **Python 3.11+** | Широко распространён, прост, богатая экосистема. |
| **FastAPI** | Высокая скорость разработки, автоматический Swagger/OpenAPI, поддержка асинхронности и встроенная инверсия зависимостей (`Depends`). |
| **Pydantic v2** | Декларативная валидация входных данных, автоматическая генерация схем. |
| **SQLAlchemy 2.0** | Зрелый ORM, типобезопасный стиль 2.0, переносимость между СУБД. |
| **SQLite** | Не требует отдельного сервера — упрощает запуск и демонстрацию; для прод‑нагрузки заменяется на PostgreSQL сменой `DATABASE_URL`. |
| **Uvicorn** | Быстрый ASGI‑сервер, стандарт для FastAPI. |
| **pytest + httpx** | Удобные модульные и интеграционные тесты (через `TestClient`). |
| **HTML/CSS/JS (vanilla)** | Минимум зависимостей, простой запуск, лёгкая поддержка для учебного проекта. |

## 7. Архитектура (обзор)

Слоистая архитектура с инверсией зависимостей (см. диаграммы в
[architecture.md](architecture.md)):

1. **routers** — принимают HTTP‑запрос, валидируют параметры, возвращают коды ответов;
2. **services** — реализуют правила предметной области (бизнес‑логику);
3. **repositories** — инкапсулируют запросы к БД;
4. **models** — ORM‑описание таблиц;
5. **schemas** — DTO и валидация (Pydantic);
6. **core** — доменные исключения и их перевод в HTTP‑ответы.

Соблюдаются принципы:
- **SRP** — каждый класс отвечает за одну задачу (репозиторий — данные, сервис — логика);
- **DIP** — сервисы зависят от репозиториев через конструктор, а не от деталей БД;
- **DRY** — общие операции (границы месяца, агрегаты) вынесены в переиспользуемые функции;
- **KISS** — простая конфигурация, отсутствие лишних абстракций.

## 8. Модель данных (обзор)

Три сущности: `categories`, `transactions`, `budgets`. Подробная схема и
ER‑диаграмма — в [database.md](database.md).

## 9. Ограничения и допущения

- Приложение **однопользовательское** (без аутентификации) — учебная область.
  Расширение до многопользовательского режима — добавлением сущности `User` и
  внешних ключей (см. раздел «Развитие»).
- Суммы хранятся как число с плавающей точкой с округлением до 2 знаков.
  Для финансово‑строгих расчётов рекомендуется тип `Decimal`/целые копейки —
  отмечено как осознанный компромисс (KISS) для учебного проекта.
- Помесячная агрегация использует функцию `strftime` (специфично для SQLite).

## 10. Критерии приёмки проекта
